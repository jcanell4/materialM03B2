Informalment podem dir que una funció és un conjunt de línies de codi identificades per un nom, de manera que el llenguatge de programació pot reconèixer el codi a través del nom. Així, la majoria de llenguatges de programació poden executar el codi d'una funció fent només referència al seu nom.

És a dir, com podeu veure a l'exemple, és equivalent escriure les tres línies de codi contingudes a la funció diguesHola que invocar el seu nom com si fos una instrucció del llenguatge.

En el context del llenguatge Java diem que un parell de claus delimiten un bloc de codi, en referencia al conjunt d'instruccions. A partir d'ara usarem indistintament conjunt de línies de codi o bloc de codi.

Què cal saber - Funcions

Definició formal

La definició formal posa l'èmfasi en el que la funció fa. Al material didàctic del mòdul podeu trobar la definició formal de funció:

En general dins dels llenguatges de programació, s’anomena funció a un conjunt d’instruccions amb un objectiu comú que es declaren de manera explícitament diferenciada dins del codi font mitjançant una etiqueta o identificador.

És important destacar d'aquesta definició, el fet que les instruccions no poden ser línies de codi agafades aleatòriament d'un programa, sinó que es troben agrupades per alguna raó específica, per aconseguir un objectiu comú o dit d'una altra manera, per realitzar una tasca concreta.

De fet, s'anomenen funcions per la seva semblança amb la paraula homònima de l'àmbit matemàtic. Des de la perspectiva matemàtica, el concepte de funció és relaciona amb una transformació ben definida d'un o més valors (entrada). La transformació dóna com a resultat un valor (sortida) que depèn directament dels valors d'entrada.

Són exemples de funcions matemàtiques les sumes, restes i en general qualsevol operació o combinació d'operacions. Algunes funcions poden arribar a ser realment complexes, però totes compleixen el requisit que sempre obtindrem el mateix resultat si fem servir les mateixes entrades.

De forma similar, les funcions dels llenguatges de programació s'han de codificar per tal aconseguir realitzar una tasca ben definida

Què cal saber - Funcions

Operativa bàsica

Un cop donada la definició, convé que us adoneu que, encara que sense saber-ho, heu estat usant funcions des de que vàreu començar aquest mòdul en el bloc 1. Això és així perquè el llenguatge Java obliga a escriure totes les instruccions dins d'alguna funció. No és possible escriure codi fora de les funcions, el llenguatge no sabria interpretar-lo. És per això que Java estableix el nom de la funció main, com una forma d'identificar el codi que cal executar en primer lloc, en iniciar un programa. D'aquesta manera JAVA aconsegueix localitzar on es troba l'inici de les aplicacions, a l'hora que manté la premissa de contenir tot el codi executable dins d'alguna funció.

Val a dir que en el llenguatge Java, les funcions s'anomenen mètodes. Segur que la paraula us sona més, però a efectes pràctics, els podeu considerar el mateix. El llenguatge Java disposa de mecanismes per definir el contingut dels mètodes, es coneixen amb el nom genèric de declaració de mètodes i permeten informar al llenguatge del codi que quedarà agrupat dins d'un mètode i de com s'intercanvia la informació generada dins i fora d'aquest.

La declaració bàsica d’un mètode es fa usant la sintaxi que es mostra tot seguit:


    public void nomMetode() {
        //Aquí dins aniran les seves instruccions
        //...
    }

La declaració de mètodes es pot dur a terme en qualsevol lloc del fitxer font sempre que sigui entre les claus que identifiquen l’inici i fi de fitxer (public class NomClasse { … }) i fora del bloc d’instruccions del mètode principal, o qualsevol altre mètode.

La declaració dels mètodes no executa el seu contingut. Per executar el codi d'un mètode, cal invocar els seu nom seguit de dos parèntesis, que són els signes que identifiquen les funcions i les distingeixen de les variables. La invocació de mètodes s'anomena també crida. Vegem-ne un exemple


    public void inici(){
        //...
        nomMetode(); //aquesta és la instrucció de crida que executarà el codi contingut a la funció "nomMetode"
        //...
    }

    public void nomMetode() {
        //Aquí dins aniran les seves instruccions
        //...
    }

A l'exemple anterior es fa una crida al mètode nomMetode com a part de les instruccions del mètode inici.

Què cal saber - Funcions

Característiques fonamentals de les funcions

Una de les principals raons de ser de les funcions, és minimitzar la complexitat de les implementacions reduint el nombre de línies de codi que un programador ha de tenir en compte, a l'hora de fer una programa.

Les funcions permeten construir noves instruccions cada cop més complexes, que es poden usar per implementar grans aplicacions escrivint un menor número de línies.

És per això que diem que les funcions encapsulen o amaguen el codi que contenen.

Certament, no és necessari conèixer el codi intern d'una funció per poder-la fer servir, només cal saber què és el que la funció fa i com i quan cal fer-la servir.

Gràcies a les funcions, és possible fer implementacions eficients en equip i reduir al mateix temps, el nombre d'errors generats durant la codificació.

A més de reduir la complexitat, les funcions també eviten la repetició de codi de forma innecessària, ja que les instruccions d'una funció només s'escriuen una única vegada.

Això ens obliga a dissenyar de forma acurada les funcions, intentat, sempre que es pugui, escriure mètodes genèrics que puguin reutilitzar-se en diferents situacions.

Malauradament, això pot acabar incrementant el nombre final de línies usades o fins i tot, acabar generant una implementació globalment més complexa. Però des del punt de vista de la implementació que cada programador acabarà fent, la complexitat es redueix sempre i per tant surt a compte l'ús de funcions.

Què cal saber - Funcions

Intercanvi d'informació. Àmbit de variables i ús de paràmetres

Si les funcions encapsulen codi, quan un programador en fa servir una, no hauria de preocupar-se de quines variables fa servir internament, ni de quines variables es manipulen internament.

Això obliga al llenguatge a treballar amb el que s'anomena àmbit de les variables. Entenem per àmbit de les variables, que en la identificació de les mateixes no només intervé el seu nom, sinó també el lloc on han estat declarades (per exemple dins una funció). D'aquesta manera és possible utilitzar el mateix nom de variables en funcions diferents sense que es produeixi cap mena d'interferència. De fet, durant l'execució seran tractades sempre com variables diferents.

Si això és així, el llenguatge haurà de disposar de mecanismes per intercanviar informació entre diferents funcions, ja que sovint poden complementar-se i treballar en cadena. El mecanisme previst pel llenguatge Java és la declaració de paràmetres (entrada) i del resultat obtingut en executar la funció (sortida), per tal d'aprofitar aquest resultat en altres llocs del codi.

La declaració de paràmetres s'afegeix a la declaració de la funció indicant el tipus de dada i un nom que els identifiqui. El resultat retornat, també es declara indicant el tipus de dada retornada. Així la declaració bàsica d'una funció quedaria de la marera següent:


tipusDadaRetorn nomFuncio(tipusParam1 nomParam, ..., tipusParamN nomParamN){
    <declaració de variables de la funció>;
    <instruccions>;
    return <dada de retorn>;
}

Fixeu-vos que d'acord amb aquest mecanisme les funcions només poden retornar un únic valor. Més endavant estudiarem com canviar aquesta limitació.

Òbviament la forma anterior de declarar funcions continua vigent. En realitat aquest nou mecanisme és una extensió de l'anterior de manera que el tipus void significa que la funció no retorna cap dada i l'absència de paràmetres declarats entre els parèntesis implica que la funció no necessita cap entrada de dades.

El mecanisme per invocar mètodes que declaren paràmetres, també canvia lleugerament, doncs a la crida caldrà especificar els valors d'entrada mitjançant una expressió del llenguatge. Si hi ha retorn (tipus diferent a void), caldrà recollir el valor retornat dins una variable per tal que no es perdi el resultat retornat. Usarem l'assignació per passar el valor retornat a la variable escollida i correctament declarada:


void inici(){
    /*declaració d'una variable del mateix tipus de retorn per recollir el resultat*/
   tipusDadaRetorn variablePerRecollirElResultat; 
   
   ...
                                    
   variablePerRecollirElResultat = nomFuncio(<expressió per el param 1>, ..., <expressió per el param n>);

   ...
}

Estructura d'un programa que fa servir diverses funcions

Abans de continuar, cal presentar un canvi necessari en el format dels vostres programes quan es vol declarar altres funcions, a més de la principal (main). Concretament, per les característiques del llenguatge Java, cal que el mètode principal tingui un format molt concret. En cas contrari, hi haurà un error de compilació en futures passes del procés.

Tot el codi que normalment posaríeu dins el bloc d’instruccions del mètode principal, cal ubicar-lo en una funció "secundària", que anomenaren inici. S'ha triat aquest nom per tal que us resulti clarificador, però podríem haver-li posat qualsevol altre nom.

Dins el mètode principal simplement s’invoca aquesta nova funció. De fet, no és imprescindible que conegueu els detalls dels motius pels quals és necessari fer aquest canvi. Simplement podeu usar el codi següent d’exemple com a plantilla per generar els vostres programes, tenint en compte que tot el codi que posaríeu normalment al mètode principal, ara anirà al mètode inici.


public class ProgramaX {

    ProgramaX programa = new ProgramaX();
    programa.inici();
  }

  public void inici() {
    //Instruccions del mètode principal (problema general)
    //...
  }
  
  //Resta de mètodes
  //...
}

En usar aquest codi com a plantilla, noteu que a la línia següent caldria posar el nom de la classe que esteu editant, en lloc de ProgramaX:


ProgramaX programa = new ProgramaX();

Què cal saber - Funcions

Ampliació de l'àmbit de les variables

La majoria de llenguatges de programació incloent JAVA, permeten definir diversos àmbits de variables, no només l'àmbit dins una funció. En Java, l'àmbit de validesa d'una variable es delimita per mitjà de les dues claus (la d'obertura i la tancament) més properes a la variable. És per això que la funció actua com un delimitador d'àmbit, doncs el codi (incloses les variables) es troba tancat entre claus

Però no és l'únic. Si declarem una variable entre les claus d'una instrucció if o while, la variable només serà valida dins del codi que es trobi entre els claus i no pas en altre lloc.

Java permet fins i tot definir àmbits a demanda del programador. Només caldria tancar la porció de codi que desitgem definir com a àmbit, entre dues claus i declarar-hi dins les variables: 


                                            public void unaFuncio(){
                                                //àmbit de funció
                                                int variableDeLaFuncio;
                                                
                                                ...
                                                
                                                if(variableDeFuncio){
                                                     //àmbit d'instrucció 
                                                     int variableDeLIf:
                                                     ...
                                                }
                                                
                                                {
                                                    //àmbit a demanda. La variable només serà
                                                    //vàlida pel codi que es trobi dins les claus
                                                    int variableValidaNomesAqui;
                                                    ...
                                                }
                                            }

L'única limitació que té aquest sistema és que les claus han de mantenir una imbricació correcte. És a dir no podem tancar una clau mentre contingui parells oberts.

Hi ha encara un àmbit major que els funcions. Fixeu-vos que quan escriviu programes, Java us obliga a definir també el nom del fitxer usant una declaració com:


                                            public class nomDelFitxer{
                                                ...
                                            }

en la que totes els funcions estan contingudes entre dues claus.

Efectivament, Java permet declarar variables de l'àmbit fitxer, declarant les variables en el mateix nivell que es declaren les funcions. Sovint aquest àmbit se sol anomenar global perquè una variable així declarada serà vàlida per ser usada des de qualsevol mètode. 


                                            public class nomDelFitxer{
                                                int variableDAmbitGlobal;
                                                
                                                void unaFuncio(){
                                                    //aquí es pot fer servir la variable variableDAmbitGlobal
                                                    ...
                                                    variableDAmbitGlobal=5;
                                                    ...
                                                }
                                        
                                            }

Per contraposició a les variables globals, les altres s'anomenen locals.

És important saber utilitzar els àmbits de les variables adequadament perquè ens poden estalviar molts maldecaps. A mida que les aplicacions que anem programant creixin, el nombre de variables a utilitzar creixerà també i ens resultarà molt difícil controlar-les. Sovint, per comoditat, estarem temptats de fer servir només variables globals, però aquesta no és un bona decisió:

  • Les variables globals ocupen espai en memòria des de que s'inicia l'execució de l'aplicació fins que es tanca. Per tant, malbaraten la memòria.
  • Un canvi de valor d'una variable global en una línia de codi, podria afectar el resultat en una altre línia de codi allunyada de la que s'ha fet el canvi, i per tant difícilment relacionable.
  • La gran major part d'aplicacions s'implementen en equip. Sovint haurem d'usar funcions que no haguem escrit nosaltres i per tant no sabrem quines variables manipula. Si per fer servir la funció cal aprendre com estar feta, perdrem tota l'eficiència i avantatges de fer servir funcions.

De fet, l'ús de variables globals només es considera adequat si hi ha una dada que ha de ser manipulada per diverses funcions, l'objectiu de les quals impliqui canvis parcials que acabin resolent un problema més gran. Malauradament, això només passa si l'aplicació és petita o només implica la resolució d'un únic problema.

Un exemple força clarificador de l'ús de variables globals podria ser la implementació d'una calculadora. El resultat que es va mostrant a la pantalla es podria declarar en una variable global, ja que les diferents operacions fan servir sempre el resultat de la pantalla per a realitzar el nou càlcul.

Una bona política de declaració de variables consisteix en reduir el nombre de variables globals a les estrictament imprescindibles, tenint en compte que en cas de dubte cal optar sempre per variables locals i declarar totes les variables locals sempre a nivell de mètode.

En cas de necessitar declarar variables en àmbits menors que les funcions, cal assegurar-se que es tracta només de variables auxiliars dins el bloc de codi on es declaren i que mai es faran servir fora del bloc.

EL nivell d'àmbit de funció és suficient per reduir la complexitat i aconseguir controlar les variables de la immensa majoria d'aplicacions. Només en cas d'haver d'implementar una funció amb moltes línies de codi, ens pot resultar recomanable fer declaracions de variables d'àmbits menors. Tingueu en compte però que això no hauria de passar mai, ja que en aquests casos és preferible dividir les funcions grans en vàries de més petites.

Exercici 3- Prioritats i il·legalitats declarant variables en diferents àmbits

Java pretén ser un llenguatge que facilita la creació d'aplicacions robustes i amb pocs errors. Bona part d'això es recolza en polítiques restrictives que intenten evitar l'error humà.

La declaració de variables amb un mateix nom pot suposar una font d'errors important. Per això Java no permet declarar variables d'un mateix nom en un mateix àmbit. Fins i tot, en alguns casos la limitació s'estén malgrat que siguin de diferent àmbit. Es tracta però d'una restricció que per raons històriques i d'usabilitat contempla diverses excepcions que cal conèixer.

A continuació se us mostra un codi amb diverses situacions en que es declaren variables de diferent àmbit amb el mateix nom. Copieu-lo al Netbeans i analitzeu quins són els casos que donen error i quins no. Seguidament comenteu les línies d'error i executeu el codi per tal de veure el comportament dels casos sense error.

Un cop analitzat el codi, llegiu les següents frases i indiqueu si es tracta d'afirmacions certes o falses.

  1. És possible declarar un paràmetre d'una funció i una variable local de la mateixa, usant el mateix nom, perquè els paràmetres es troben en una àmbit diferent al de les variables locals
  2. Java no imposa restriccions a l'hora de permetre declarar variables locals que tinguin el mateix nom que variables globals, ja que en una invocació, la variable local tindria sempre prioritat sobre la global
  3. Java no permet declarar variables globals que tinguin el mateix nom que algun paràmetre d'alguna funció
  4. Java no permet declarar una variable en un bloc de codi si aquest es troba declarat en un altre bloc que contingui declarada una variable amb el mateix nom, a excepció que es tracti d'una variable global
  5. Java permet declarar variables amb noms iguals si pertanyen a àmbits diferents que no es continguin, com per exemple en dues funcions diferents perquè mai hi haurà interferències
  1. L'afirmació és falsa. Java no permet declarar un paràmetre d'una funció i una variable local de la mateixa usant el mateix nom, doncs no hi ha manera de distingir-les en invocar-les. De fet l'àmbit dels paràmetres i el de les variables locals de funció són idèntics
  2. L'afirmació és certa.
  3. L'afirmació és falsa. Java permet declarar paràmetres amb el mateix nom que les variables globals. De fet, les restriccions dels paràmetres són les mateixes que les de les variables locals de funció i per tant, en cas d'anomenar-se igual, el paràmetre tindria prioritat sobre la variable global
  4. L'afirmació és parcialment certa, ja que Java permet declarar variables amb el mateix nom si pertanyen a blocs diferents malgrat que un estigui contingut dins l'altre sempre i quant la variable del més gran es trobi declarar posteriorment al bloc menor, ja que en sortir del bloc menor la seva variable deixa d'existir i és possible fer una declaració nova amb el mateix nom.
  5. L'afirmació és certa.
